The Tech

QuantPolitik is built on a polyglot platform of Postgres, Python (Pandas, Django), and JavaScript, and is hosted in the cloud, courtesy of Heroku.

  • View the Sources

The Database (Postgres, SqlAlchemy, Redis):

The database is built in Postgres, featuring 147 variables in 9 tables, with 10 additional tables that automatically update and store new values, but more on that later. Each morning, at 7 am, a worker program, written in Python using the RQ library, launches a Redis server that runs 3 programs sequentially.

  • Sources

The Quantification (Requests, Regex, SqlAlchemy):

The first, “sec_state_schedule_quantifier,” goes the State Department’s website, finds the index of every day’s schedule, using a custom built, regex determines which is today’s schedule, and downloads the Secretary of State’s schedule using the requests library. In memory, the program then begins parsing the Secretary’s schedule to determine who he met with (Prime Ministers, Presidents, Foreign Ministers, etc) and from which country that emissary was from. The program then tallies every country that had a meeting, and assigns a point value for the interaction based on who was in attendance (2 points for a Head of State/Government, 1 for a minister). The program having quantified the Secretary’s daily schedule then takes its data and autonomously updates the Postgres database, using the SqlAlchemy library, and SQL commands for all database interaction. The queue then launches another program that does the same lexical quantification, but for the President and the Vice President.

The Formula (SqlAlchemy, Pandas, Numpy, Stats):

After both programs have run, another program, one that applies my QuantPolitik formula. This program pulls all of my data (147 columns * 198 countries), applies and munges the data using Pandas to organize it and numpy in addition stats to apply the necessary mathematics upon it. Having applied the math to the new data, the program updates my database once more, this time with its new score.

The Server (Django):

My server is built upon Django’s MVC framework and powers every webpage on the site. While in testing, QuantPolitik relied upon Flask, the decision to move to Django was necessitated by the Heroku platform, and has increased the modularity and speed as a result.

  • Sources

The API (Django, SqlAlchemy, and pure Python):

The API which accesses my database is written as a “view” within my Django framework and relies on SqlAlchemy to access the database and deliver customized results based on API received. The API is able to apply this customizability using a regex that parses the request and passes it along to the database. Once the database is accessed, the results are formatted into JSON, and passed by the API to a specifically built page based on the request submitted.

The Maps (D3):

The maps on the site are the major users of the API, (though there are other features that do use them), on the data is propagated in JSON and returned view the API, the choropleth mapping program, retrieves the data and using the D3 library builds SVGs representing a map of the world, and colors the map based on the quantized-scale of the data received. The D3 map is meant to be scalable and portable as the data constantly changes, yet the scales are redrawn accordingly and the colors consistently shift.

  • Sources

The Site (HTML5, CSS3, JavaScript):

The rest of the site, which holds and supports the Maps, relies on a standard mix of HTML5, CSS3, and where appropriate-JavaScript. Aspects of the website are mobile optimized, and all work properly on a desktop. The pages themselves were inspired by html5up.net, but all feature significant customization and personalization (everything else-except the D3 map coordinates-was built from scratch by myself). It should also be noted that the media is all open source from Wikimedia. The dashboard-while initially inspired by a free design from html5up.net, was engineered specifically to meet the needs of hosting a choropleth map visualization and for accessing each data table independently, and each column within each table.

The Cloud (Heroku):

While QuantPolitik was initially hosted on Amazon Web Servers, the fact that the “hobby” or lowest tier still wound up costing $20 a month, whereas the Heroku bottom tier was free, it thus made the switch quite sensible from the financial perspective. However, Heroku’s bottom tier does not provide for the dyno to continuously run, so during periods of inactivity, accessing the web page takes considerably longer than preferable due to the dyno “going to sleep.” But since it’s free, it’s hard to complain. That said; Heroku hosts my django server, my redis server, my queue, and my entire Postgres database, the site is being continuously updated and deployed to Heroku as needed.
